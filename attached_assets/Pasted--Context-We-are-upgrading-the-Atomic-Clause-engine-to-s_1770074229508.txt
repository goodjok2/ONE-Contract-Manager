## Context
We are upgrading the "Atomic Clause" engine to support advanced features from a legacy version (`wizardv3`).
**Current State:** We have the tables (`home_models`, `projects`) but lack the "Glue" logic that connects them to the contract generation.

## Objective
Port the **Component Library** (Dynamic Table Generation) and **Variable Mapping System** logic from `wizardv3` to the current codebase.

## Requirements

### 1. Component Library (Dynamic Tables)
* **Goal:** Allow clauses to contain tags like `{{PAYMENT_SCHEDULE_TABLE}}` which render as HTML tables based on project data.
* **Implementation:**
    * Create `server/services/component-library.ts`.
    * Implement functions to generate HTML tables for:
        * `renderPaymentSchedule(contractValue)`
        * `renderUnitPricingTable(units)`
        * `renderExhibitsList(exhibits)`
    * **Integration:** Update the `ClauseRenderer` (or equivalent text processing function) to detect these `{{TAGS}}` and replace them with the output of these functions.

### 2. Variable Mapping Logic
* **Goal:** specific database fields (e.g., `llc.name`, `project.address`) must map to variable tokens (e.g., `{{seller_name}}`, `{{site_address}}`).
* **Implementation:**
    * Create `server/services/variable-mapper.ts`.
    * Create a function `getContractVariables(projectId, contractId)` that:
        1. Fetches the Project, Units, Home Models, and LLC data.
        2. Flattens them into a `key: value` map.
        3. Returns the map for use by the Clause Renderer.

### 3. Auto-LLC Generation Logic
* **Goal:** Logic to automatically suggest an LLC name based on the project address.
* **Implementation:**
    * Create a helper `generateLLCName(address, state)` in `server/services/llc-service.ts`.
    * **Logic:**
        1. Extract the street name (e.g., "123 Main St" -> "Main").
        2. Append "Project, LLC" (e.g., "Main Project, LLC").
        3. Check DB for uniqueness. If taken, append a number ("Main Project II, LLC").
    * **API:** Expose this via `POST /api/llcs/suggest-name`.

## Technical Specifications
* **Code Style:** TypeScript, Async/Await.
* **Compatibility:** Ensure the HTML generated by the Component Library uses standard inline styles suitable for PDF generation (if applicable) or standard Tailwind classes if web-only.

## Constraints
* **Do NOT** overwrite the existing atomic clause structure. This is an *additive* layer to process the *content* of the clauses.
* **Preserve** the multi-tenant `organizationId` checks in all data fetching.
